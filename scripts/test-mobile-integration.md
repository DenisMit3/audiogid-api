# Тестирование интеграции мобильного приложения

## Подготовка

1. Убедиться, что backend API доступен по адресу `http://82.202.159.64/v1`
2. Запустить мобильное приложение в режиме отладки
3. Включить логирование сетевых запросов

## Проверка конфигурации API

### В коде приложения

Проверить файл `apps/mobile_flutter/lib/core/config/app_config.dart`:

```dart
// Должно быть:
const cloudUrl = 'http://82.202.159.64/v1';
```

### В runtime

В приложении проверить:
1. Открыть DevTools или логи
2. Найти сообщения о подключении к API
3. Убедиться, что baseUrl правильный

## Тест-кейсы для ручного тестирования

### 1. Инициализация и выбор города

**Шаги:**
1. Запустить приложение
2. Пройти онбординг (если первый запуск)
3. Выбрать город (kaliningrad_city или kaliningrad_oblast)

**Ожидаемый результат:**
- Города загружаются из API
- Выбранный город сохраняется
- Переход на главный экран

**Проверка в логах:**
```
[DEBUG] GET http://82.202.159.64/v1/public/cities
[DEBUG] Response: 200 OK
```

### 2. Загрузка списка туров

**Шаги:**
1. Открыть экран "Туры"
2. Дождаться загрузки

**Ожидаемый результат:**
- Список туров отображается
- Карточки туров показывают обложки, названия, метаданные
- Skeleton loader показывается во время загрузки

**Проверка в логах:**
```
[DEBUG] GET http://82.202.159.64/v1/public/catalog?city=kaliningrad_city
[DEBUG] Response: 200 OK
[DEBUG] Tours synced to local DB
```

**Проверка в БД:**
- Открыть DevTools → Database Inspector
- Проверить таблицу `tours` в Drift БД
- Должны быть записи с `city_slug = 'kaliningrad_city'`

### 3. Открытие деталей тура

**Шаги:**
1. Нажать на карточку тура
2. Открыть TourDetailScreen

**Ожидаемый результат:**
- Детали тура загружаются
- Отображается описание, точки маршрута, кнопки действий
- Tour Timeline отображается (если реализован)

**Проверка в логах:**
```
[DEBUG] GET http://82.202.159.64/v1/public/tours/{tourId}
[DEBUG] Response: 200 OK
```

### 4. Загрузка каталога POI

**Шаги:**
1. Открыть экран "Каталог"
2. Дождаться загрузки

**Ожидаемый результат:**
- Список POI отображается
- Фильтры работают
- Поиск работает

**Проверка в логах:**
```
[DEBUG] GET http://82.202.159.64/v1/public/catalog?city=kaliningrad_city
[DEBUG] Response: 200 OK
[DEBUG] POIs synced to local DB
```

### 5. Открытие деталей POI

**Шаги:**
1. Нажать на карточку POI
2. Открыть PoiDetailScreen

**Ожидаемый результат:**
- Детали POI загружаются
- Отображаются изображения, описание, narrations
- Кнопки "Слушать", "Превью", "Скачать" работают

**Проверка в логах:**
```
[DEBUG] GET http://82.202.159.64/v1/public/poi/{poiId}
[DEBUG] Response: 200 OK
```

### 6. ETag кэширование

**Шаги:**
1. Загрузить данные (туры/POI)
2. Проверить сохранение ETag в БД
3. Сделать повторный запрос

**Ожидаемый результат:**
- Первый запрос возвращает данные и ETag
- ETag сохраняется в таблице `etags` в Drift БД
- Второй запрос с `If-None-Match` возвращает 304
- Данные берутся из кэша

**Проверка в логах:**
```
[DEBUG] First request: GET ... (Status: 200, ETag: "abc123")
[DEBUG] ETag saved to DB
[DEBUG] Second request: GET ... (If-None-Match: "abc123", Status: 304)
[DEBUG] Using cached data
```

### 7. Offline режим

**Шаги:**
1. Загрузить данные при наличии интернета
2. Отключить интернет (Airplane Mode)
3. Открыть приложение

**Ожидаемый результат:**
- Данные отображаются из локальной БД
- Нет ошибок сети
- UI показывает, что данные из кэша

**Проверка:**
- Данные должны быть в Drift БД
- Запросы к API не должны выполняться

### 8. Аудио плеер

**Шаги:**
1. Выбрать тур с доступом
2. Нажать "Начать тур" или "Слушать"
3. Проверить работу плеера

**Ожидаемый результат:**
- Аудио загружается и воспроизводится
- Mini player появляется
- Play/Pause/Next/Previous работают
- Прогресс отображается

**Проверка в логах:**
```
[DEBUG AUDIO] loadPlaylist called: tourId=..., items=...
[DEBUG AUDIO] Queue built: X items
[DEBUG AUDIO] Play started
```

### 9. Entitlements и покупки

**Шаги:**
1. Проверить текущие entitlements
2. Попытаться купить тур (если настроен IAP)
3. Проверить обновление entitlements

**Ожидаемый результат:**
- Entitlements загружаются из API
- Покупка проходит через IAP
- После покупки entitlement появляется в списке
- UI обновляется (бейджи "Открыто")

**Проверка в логах:**
```
[DEBUG] GET http://82.202.159.64/v1/billing/entitlements?device_anon_id=...
[DEBUG] POST http://82.202.159.64/v1/billing/apple/verify (или google/verify)
[DEBUG] Entitlements synced
```

### 10. Tour Mode

**Шаги:**
1. Запустить тур
2. Включить геолокацию
3. Проверить работу Tour Mode

**Ожидаемый результат:**
- Карта отображается с маршрутом
- Текущая позиция отображается
- Auto-play работает (если включен)
- Off-route detection работает

**Проверка в логах:**
```
[DEBUG] Tour Mode started: tourId=...
[DEBUG] Location updated: lat=..., lon=...
[DEBUG] Geofence entered: poiId=...
[DEBUG AUDIO] Auto-play triggered
```

## Проверка ошибок

### Сетевые ошибки
1. Отключить интернет
2. Попытаться загрузить данные
3. **Ожидаемый результат**: Показывается error state или данные из кэша

### Ошибки API
1. Отправить невалидный запрос
2. **Ожидаемый результат**: Показывается понятное сообщение об ошибке

### Ошибки авторизации
1. Использовать невалидный токен
2. **Ожидаемый результат**: Редирект на логин или показ ошибки
